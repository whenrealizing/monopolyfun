# 交易链路测试交接

更新时间：2026-05-20

## 目标

用两个普通账号覆盖 MonopolyFun 的核心交易链路，确认发布、下单、付款、交付、验收、争议和最终状态都能按预期推进。

建议账号：

- 账号 A：发布方 / 买家 / 需求方 / 付款方
- 账号 B：接单方 / 卖家 / 执行方 / 收款方

通用检查项：

- 首页/探索列表能看到新发布的卡片，筛选只影响探索列表，不影响热门卡片。
- 创建订单后，双方都能在“我的订单”看到订单；无关账号不能进入订单详情。
- 订单详情顶部阶段、订单摘要、结算区、事件记录、下一步操作一致。
- money 订单必须关注 `paymentIntentStatus` 和 `paymentDueAt`：列表、订单摘要、订单详情都应该展示一致状态。
- 订单最终只应进入 `final_accepted` 或 `final_closed`，不应出现按钮还能继续重复操作的情况。

## 链路 1：售卖内容，人工交付，OKX 付款，正常验收

目的：覆盖最标准的“买家购买卖家商品/服务”链路。

前置：

- 账号 B 已登录并能发布供给。
- 账号 A 有可支付的钱包环境；如果是本地假支付，确保 fake callback 或测试支付入口可用。
- 如果真实钱包付款时报 `Insufficient balance: available=0, required=10000`，先确认账号 A 钱包在订单要求的链上持有对应资产。订单要求 `USDC` 时，钱包只有 `USDT` 会被视为 USDC 可用余额为 0。
- 如果创建 OKX 支付会话时报 `java.lang.NullPointerException`，栈指向 `PaymentService.mapValue -> normalizeProviderOkxReconciliation -> okxMetadata`，说明 OKX 返回的 reconciliation 子对象里包含 null value。已修复为忽略子 Map 中的 null value；复测时应确认接口返回业务错误或支付会话，而不是 500。

步骤：

1. 用账号 B 进入发布页，发布一个“供给 / 售卖内容”。
2. 结算方式选择 OKX / money。
3. 交付方式选择人工交付或需要证明的普通交付，不要选择直接发货或库存托管。
4. 填写交付要求、验收标准、价格，发布成功。
5. 切到账号 A，在首页/探索中找到该卡片，点击进入详情。
6. 选择售卖内容，点击“先付款购买 / 立即购买”。
7. 进入订单页，确认订单状态为 `claimed`，支付状态为 `missing` 或 `pending`，并能看到付款截止时间。
8. 用账号 A 创建支付会话并完成付款。
9. 刷新支付状态，预期支付状态变为 `captured / 已付款`。
10. 切到账号 B，进入该订单，提交交付证明：摘要不少于 8 个字符，至少上传 1 个附件。
11. 切回账号 A，确认订单进入 `delivered / 待验收`，能看到交付证明。
12. 账号 A 点击“确认通过”。
13. 确认订单进入 `accepted_open`，争议窗口开放。
14. 等待或触发争议窗口结束后，确认订单最终进入 `final_accepted`。

预期：

- 付款前，交付按钮应被支付状态拦住。
- 付款成功后，卖家才能提交交付。
- 验收后结算区显示 money 已完成，支付状态为已付款。
- 列表中的订单摘要展示 `paymentIntentStatus=captured`，不再显示待付款。

完成记录：

- 状态：未完成
- 订单号：
- 发现问题：
  - 付款时报错 `Insufficient balance: available=0, required=10000`。初步判断为账号 A 钱包没有订单要求的 USDC；只有 USDT 不能支付 USDC intent。
  - 创建/提交 OKX 支付时出现 `java.lang.NullPointerException`，栈在 `PaymentService.mapValue`。已做后端窄修复，需复测确认不再返回 500。

## 链路 2：需求接单，发布人付款，执行方交付，正常验收

目的：覆盖 request 场景，确认付款方不是接单人，而是需求发布人。

前置：

- 账号 A 作为需求发布人。
- 账号 B 作为执行方，需要准备一个有效 OKX / X Layer USDC 收款地址。

步骤：

1. 用账号 A 发布一个“需求 / request”，新增一个可接任务。
2. 结算方式选择 OKX / money。
3. 填写交付要求、验收标准、金额。
4. 切到账号 B，进入该需求详情。
5. 选择任务，填写执行备注；如页面要求，填写收款钱包地址。
6. 点击“立即接单”。
7. 进入订单页，确认订单已生成，当前应提示等待发布人付款。
8. 切到账号 A，进入该订单，创建支付会话并完成付款。
9. 切回账号 B，刷新订单，确认支付状态已付款，提交交付证明。
10. 切到账号 A，查看证明并点击“确认通过”。
11. 确认订单进入 `accepted_open`，最终进入 `final_accepted`。

预期：

- 订单角色文案应显示：账号 A 是付款方/发布人，账号 B 是交付方/执行方。
- 账号 B 不应能替账号 A 完成付款。
- 账号 A 付款完成前，账号 B 的交付动作应不可用或被明确提示拦截。
- 订单摘要、列表、详情支付状态一致。

## 链路 3：股份结算，无 OKX 付款，正常验收

目的：覆盖 shares 结算，不创建 payment intent，不出现 OKX 付款入口。

前置：

- 账号 A 发布一个项目任务或股份结算任务。
- 账号 B 作为执行方。

步骤：

1. 用账号 A 发布一个股份结算任务。
2. 结算方式选择 shares / 股份。
3. 填写股份数量、交付要求、验收标准。
4. 切到账号 B 接单。
5. 进入订单页，确认没有“创建支付会话 / 使用钱包支付”入口。
6. 账号 B 提交进度，确认事件记录出现进度更新。
7. 账号 B 提交交付证明，至少上传 1 个附件。
8. 账号 A 查看证明并确认通过。
9. 确认订单进入 `accepted_open`，结算区显示股份处于争议窗口或待审批状态。
10. 争议窗口结束或审批完成后，确认进入 `final_accepted`，股份状态显示已释放/已审批/免审批中的合理状态。

预期：

- 全链路不应生成 OKX payment intent。
- 订单摘要的 `paymentIntentStatus` 应为空、缺省或不展示为待付款。
- 股份结算区应说明当前锁定原因，例如争议窗口、审批中或等待发放。

## 链路 4：直接发货，付款后自动发货

目的：覆盖 `instant_fulfillment` 或直接发货模式，确认付款成功后系统自动写入交付结果。

前置：

- 账号 B 发布供给。
- 账号 A 作为买家。

步骤：

1. 用账号 B 发布一个供给，交付方式选择直接发货。
2. 填写直接发货需要的配置、交付 SLA、验收标准、价格。
3. 切到账号 A，购买该供给。
4. 如页面要求，填写手机号，例如测试手机号。
5. 进入订单页，确认付款前提示“付款成功后系统会自动发货”。
6. 账号 A 完成 OKX 付款。
7. 刷新订单，确认支付状态为 `captured`。
8. 确认订单出现发货结果或 delivery receipt。
9. 账号 A 查看交付结果，确认通过。
10. 确认订单最终进入 `final_accepted`。

预期：

- 付款前不应提前暴露完整交付结果。
- 付款成功后不需要账号 B 手动上传证明，系统应自动生成交付记录。
- 如果自动发货失败，应出现“重试发货”入口，并且错误信息可理解。

## 链路 5：平台库存托管，付款后 reveal 库存

目的：覆盖 `stock_fulfillment`，确认库存锁定、付款后 reveal、库存消耗都正常。

前置：

- 账号 B 发布供给。
- 准备至少 2 条测试库存内容，避免只测单库存。

步骤：

1. 用账号 B 发布一个供给，交付方式选择平台库存托管。
2. 输入多条库存内容，发布数量应等于库存条数。
3. 切到账号 A，购买其中 1 件。
4. 进入订单页，确认付款前只显示库存预览或“已托管”，不显示完整库存。
5. 完成付款。
6. 刷新订单，确认支付状态为 `captured`，订单出现库存交付快照。
7. 点击“查看完整内容 / reveal”，确认能看到完整库存 payload。
8. 账号 A 确认通过。
9. 回到原供给详情，确认剩余库存数量减少 1。
10. 再用账号 A 或另一个测试买家购买第二件，确认不会拿到同一条库存。

预期：

- 付款前完整库存不可见。
- 每个订单只锁定并 reveal 一条库存。

## 链路 6：争议链路，交付后发起争议并处理

目的：覆盖异常路径：交付后不验收，发起争议，撤回或平台处理。

建议至少跑两次：

- 6A：发起争议后撤回，回到原验收路径。
- 6B：发起争议后由平台处理，分别验证判定交付通过和关闭订单。

步骤 6A：争议撤回

1. 任选链路 1 或链路 2 创建一个 money 订单，推进到 `delivered / 待验收`。
2. 付款方账号打开订单页，点击“发起争议”。
3. 填写争议原因，不少于 8 个字符；可附加证据链接或附件。
4. 确认订单进入 `disputed`。
5. 在争议详情页确认能看到发起人、争议原因、交付证据、争议证据和时间线。
6. 用争议发起人点击“撤回争议”，填写撤回原因。
7. 确认订单回到发起争议前的验收路径，通常是 `delivered` 或 `accepted_open`。
8. 继续确认通过，最终进入 `final_accepted`。

步骤 6B：平台处理

1. 再创建一个订单并推进到 `delivered`。
2. 付款方发起争议。
3. 用有后台/评审权限的账号进入订单或争议详情。
4. 如需要，先分配处理人。
5. 选择“判定交付通过”，确认订单进入 `final_accepted`。
6. 再创建一笔争议订单，选择“关闭订单”，确认进入 `final_closed`。

预期：

- 争议中原验收、交付、结算动作应被冻结。
- 撤回争议后订单应回到争议前状态，不应直接最终完成。
- 平台判定通过时，结算继续推进；平台关闭时，订单最终关闭，money 订单应出现退款、争议或关闭后的合理支付/结算状态。
- 争议详情页证据、时间线、处理结果都应保留。

## 支付异常补充检查

这些不是单独链路，但建议在链路 1 或链路 2 中穿插验证：

- 创建支付会话后不付款：订单列表和详情应显示待付款，并展示付款截止时间。
- 钱包取消签名：支付状态应保持 pending/missing 或变成 cancelled，页面允许重新发起。
- 支付失败：应显示失败原因；如果状态为 failed/cancelled，应允许重新生成支付要求或重新支付。
- 已最终关闭后收到迟到支付回调：后端应拒绝结算，支付状态不应把订单重新推进。

## 后续 AI 接手建议

1. 先执行链路 1，拿到一个确定可跑通的基准订单。
2. 每跑完一条链路，把订单号、账号、截图路径、发现问题补到本文对应“完成记录”。
3. 如果发现问题，先记录复现步骤和实际/预期，再决定是否修。
4. 简单样式或文案修改不要跑 `check/build`；涉及交易状态、接口或订单权限时再跑对应后端/前端测试。
